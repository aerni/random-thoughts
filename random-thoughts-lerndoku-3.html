1 Was ist dieses Projekt?

Random Thoughts ist ein einfacher Blog, den du von Grund auf selbst aufbaust. Du lernst dabei zwei Phasen kennen:

🖼️
Phase 1 — Statisch
Du setzt das Design aus Figma mit reinem HTML und Tailwind CSS um. Alle Texte, Bilder und Daten sind direkt im Code hartkodiert.
Branch: static
⚡
Phase 2 — Dynamisch
Du bindest Statamic an. Inhalte kommen aus dem CMS — du pflegst sie im Control Panel, nicht mehr im Code.
Branch: main
💡

Die fertige statische Referenzversion ist unter http://random-thoughts.test/static/blog und /static/post erreichbar — als Vergleich wenn du nicht weiterkommst.

3 Laravel Herd — Lokal entwickeln

Wenn du eine Webseite im Internet aufrufst, liegt sie auf einem Server irgendwo auf der Welt — ein Computer, der rund um die Uhr läuft und auf Anfragen reagiert. Während du eine Webseite entwickelst, willst du sie aber nicht gleich ins Internet stellen. Du brauchst also einen Server direkt auf deinem eigenen Computer — einen lokalen Server.

Genau das macht Laravel Herd: Es verwandelt deinen Mac in einen solchen lokalen Webserver, speziell für PHP-Projekte wie Laravel und Statamic.

Was Herd konkret macht

🐘
PHP bereitstellen
PHP ist die Programmiersprache hinter Laravel und Statamic. Herd installiert und verwaltet PHP automatisch — du musst es nicht selbst einrichten.
🌐
Webserver einrichten
Herd betreibt im Hintergrund einen Nginx-Webserver. Er nimmt Browser-Anfragen entgegen und leitet sie ans richtige Projekt weiter.
🔗
Domains vergeben
Für jeden Projektordner in ~/Herd erstellt Herd automatisch eine lokale .test-Adresse — die nur auf deinem Computer funktioniert.
🔒
HTTPS lokal
Herd kann lokale SSL-Zertifikate erstellen, damit auch https:// lokal funktioniert — genau wie auf einem echten Server.

Wie die .test-URL entsteht

Der Ordnername wird direkt zur URL. Herd überwacht ~/Herd dauerhaft im Hintergrund und reagiert sofort wenn ein neuer Ordner dazukommt:

Du legst an
~/Herd/
random-thoughts
→
Herd erkennt
Ordnername =
Domain
→
Du öffnest
random-thoughts
.test
→
Browser zeigt
deine lokale
Webseite

Das funktioniert, weil Herd deinem Mac beibringt, alle .test-Adressen intern aufzulösen — sie zeigen nie ins Internet, sondern immer auf deinen eigenen Computer.

Was passiert, wenn du die URL aufrufst?

Hinter jedem Aufruf von http://random-thoughts.test läuft folgendes ab:

  1. Browser schickt eine Anfrage an random-thoughts.test
  2. Herd fängt sie ab — weil .test-Adressen lokal registriert sind, landen sie nie im Internet
  3. Herd leitet weiter an den Nginx-Webserver, der den Ordner ~/Herd/random-thoughts kennt
  4. Laravel/Statamic verarbeitet die Anfrage mit PHP — welche Seite soll angezeigt werden? Welche Daten aus dem CMS?
  5. HTML wird zurückgegeben und der Browser zeigt die fertige Seite an

Vergleich: früher vs. mit Herd

Früher (manuell)Mit Laravel Herd
PHP selbst herunterladen und konfigurieren PHP ist automatisch dabei und aktuell
Apache oder Nginx manuell einrichten Webserver läuft sofort im Hintergrund
Jedes Projekt manuell als Domain eintragen Ordner hinlegen = Domain fertig
Konflikte zwischen PHP-Versionen lösen Herd verwaltet mehrere PHP-Versionen
Stundenlanger Setup für jedes neue Projekt Klonen → sofort im Browser öffnen

Die .env-Datei konfigurieren

Die .env-Datei ist die Konfigurationsdatei deiner App. Sie enthält Einstellungen die sich je nach Umgebung (lokal vs. live) unterscheiden — unter anderem die URL. Sie wird nicht mit Git synchronisiert, weil sie auf jedem Computer anders ist.

# 1. Beispieldatei kopieren (.env.example ist im Repo, .env nicht)
cp .env.example .env

# 2. In .env diese Zeile anpassen:
APP_URL=http://random-thoughts.test

# Weitere wichtige Einstellungen in .env:
APP_NAME="Random Thoughts"
APP_ENV=local          # local = Entwicklungsmodus
APP_DEBUG=true         # Fehlermeldungen anzeigen
📁

Das Projekt muss im Ordner ~/Herd liegen, damit Herd es erkennt. An einem anderen Ort (z.B. ~/Desktop oder ~/Documents) funktioniert die .test-URL nicht.

🔄

Herd läuft als Hintergrundprozess — du startest es einmal und es ist immer aktiv, solange dein Mac läuft. Du musst es nicht für jedes Projekt neu starten.

4 Git & Tower

Git ist das Versionsverwaltungs-Werkzeug, das alle Codeänderungen verfolgt und speichert. Git Tower ist eine grafische Oberfläche dazu — du siehst Branches, Commits und den Verlauf ohne Terminal-Kenntnisse.

Die wichtigsten Konzepte

BegriffWas es bedeutet
Repository Der Projekt-Ordner mit dem gesamten Code und der vollständigen Versionsgeschichte
Branch Ein paralleler Arbeitszweig — du kannst neue Dinge ausprobieren, ohne den Hauptstand zu verändern
Commit Ein Schnappschuss deiner Änderungen mit einer Beschreibung — wie ein «Speicherpunkt»
Clone Das Herunterladen eines Repositories auf deinen Computer
Push / Pull Änderungen auf GitHub hochladen (push) oder von dort herunterladen (pull)

Branches in diesem Projekt

🌱
Branch: static
Dein Startpunkt. Hier baust du das Figma-Design statisch nach. Die Template-Dateien blog.antlers.html und post.antlers.html sind leer und warten auf dich.
Startpunkt
✅
Branch: main
Der fertige Referenzstand mit allem korrekt umgesetzt — als Hilfe wenn du nicht weiterkommst oder etwas vergleichen möchtest.
Referenz

Typischer Workflow mit Tower

  1. Repo klonen: In Tower «Clone» wählen → URL eingeben → Zielordner ~/Herd setzen
  2. Branch wechseln: Links auf «Branches» → static doppelklicken
  3. Code schreiben in VS Code
  4. Änderungen committen: Dateien in Tower «stagen» → Nachricht eingeben → «Commit»

6 Antlers — Statamics Template-Sprache

Antlers ist die Template-Sprache von Statamic. Dateien mit der Endung .antlers.html sind normales HTML — ergänzt mit doppelten geschweiften Klammern {{ }} für dynamische Werte aus dem CMS.

Variablen ausgeben

<!-- Statisch (hartkodiert): -->
<h1>Blog</h1>

<!-- Mit Antlers (Wert kommt aus dem CMS): -->
<h1>{{ title }}</h1>

<!-- Mit Fallback-Wert falls kein Titel vorhanden: -->
<title>{{ title ?? site:name }}</title>

Partials — wiederverwendbare Bausteine

Ein Partial ist eine ausgelagerte HTML-Datei, die du an beliebig vielen Stellen einbinden kannst. Statt denselben Code zu wiederholen, definierst du ihn einmal als Partial — und rufst ihn dann überall auf, wo du ihn brauchst.

Das Schlüsselwort dazu ist partial: gefolgt vom Dateipfad relativ zum views-Ordner:

<!-- Syntax: {{ partial:PFAD/DATEINAME Parameter... }} -->
{{ partial:components/card
    image="/assets/bild.jpg"
    title="Mein Titel"
    date="15. April 2025"
    url="/posts/mein-post"
}}

Was bedeutet partial:components/card genau?

Dieser Aufruf lässt sich Stück für Stück lesen:

TeilWas er bedeutet
{{ }} Antlers-Tag — alles darin ist Antlers-Syntax, kein HTML
partial: Antlers-Befehl: «Binde eine Partial-Datei ein»
components/ Unterordner innerhalb von resources/views/ — dort liegen die wiederverwendbaren Bausteine
card Dateiname ohne Endung — Antlers sucht nach card.antlers.html
image="..." etc. Parameter die an das Partial übergeben werden — innerhalb der Datei als {{ image }} verfügbar

Antlers schaut also in den Ordner resources/views/components/ und lädt dort die Datei card.antlers.html. Die übergebenen Parameter werden zu lokalen Variablen innerhalb dieser Datei.

Partials im Projekt — Übersicht

Dieses Projekt hat drei Partials im Ordner resources/views/components/:

🦸
components/hero
Der grosse Titelbereich oben auf der Blog-Seite mit Überschrift und Beschreibungstext.
{{ partial:components/hero
    title="Blog"
    description="Beschreibung..."
}}
🃏
components/card
Eine Blog-Vorschau-Karte mit Bild, Titel, Datum und Link. Wird für jeden Post einmal aufgerufen.
{{ partial:components/card
    image="/assets/..."
    title="Titel"
    date="15. April"
    url="#"
}}
📰
components/article-hero
Der Titelbereich für einen einzelnen Blog-Post — wie hero, aber mit Datum und anderen Grössen. Wird in post.antlers.html verwendet.

Wie Parameter durch das Partial fliessen

Das Wichtigste zu verstehen: Was du beim Aufruf als Parameter mitgibst, landet 1:1 als Variable in der Partial-Datei. Ein konkretes Beispiel mit dem Hero-Partial:

📄 blog.antlers.html — Aufruf
{{ partial:components/hero
    title="Blog"
    description="This is a description
      of the type of content..."
}}
📄 components/hero.antlers.html — Datei
<section>
  <div class="...">
    <h1>{{ title }}</h1>
    <!-- wird zu: <h1>Blog</h1> -->
    <p>{{ description }}</p>
    <!-- wird zu: <p>This is a...</p> -->
  </div>
</section>

Ohne Partials vs. mit Partials

❌ Ohne Partials (Wiederholung)
<!-- Jede Card manuell ausschreiben -->
<article>
  <a href="#">
    <img src="/assets/bild1.jpg">
    <h2>Titel Eins</h2>
    <p>15. April 2025</p>
  </a>
</article>
<!-- × 5 wiederholt —
  Änderung am Design?
  5x anpassen! -->
✅ Mit Partials (sauber)
{{ partial:components/card
    image="/assets/bild1.jpg"
    title="Titel Eins"
    date="15. April 2025"
    url="#"
    class="md:col-span-2"
}}

{{ partial:components/card
    image="/assets/bild2.jpg"
    title="Titel Zwei"
    date="15. April 2025"
    url="#"
}}
<!-- Design ändern? Nur 1x
  in card.antlers.html! -->
🔧

Wenn du das Design einer Card ändern willst — z.B. eine andere Schriftgrösse oder Rundung — machst du das nur in card.antlers.html. Die Änderung gilt dann automatisch für alle 5 Cards auf der Blog-Seite.

Das Card-Partial aus dem Projekt

So sieht resources/views/components/card.antlers.html vollständig aus:

<article class="{{ class }}">
    <a href="{{ url }}"
       class="flex flex-col h-full overflow-hidden transition ease-out
              bg-white shadow-lg rounded-xl hover:-translate-y-1 hover:shadow-xl">
        <img src="{{ image }}"
             alt="{{ image:alt }}"  <!-- :alt = Alt-Text aus dem Asset-Manager -->
             class="object-cover w-full h-72 bg-hot-pink">
        <div class="flex flex-col justify-between flex-1 gap-4 p-6 md:gap-5 md:p-8">
            <h2 class="font-sans text-xl font-semibold md:text-2xl">{{ title }}</h2>
            <p class="font-sans text-xs">{{ date }}</p>
        </div>
    </a>
</article>

Jede {{ Variable }} darin wird beim Rendern durch den Wert ersetzt, der beim Aufruf übergeben wurde. Das {{ class }} kommt z.B. aus dem Parameter class="md:col-span-2" — damit kann die erste Card breiter dargestellt werden als die anderen.

Das Layout-System

Statamic hat ein hierarchisches Layout-System. Das Haupt-Layout (layout/main.antlers.html) umhüllt alle Seiten. Der Platzhalter {{ template_content }} markiert wo der jeweilige Seiteninhalt eingefügt wird:

<!-- resources/views/layout/main.antlers.html -->
<main class="grid gap-16 py-10 md:gap-20 md:py-20 lg:gap-24">
    {{ template_content }}
    <!-- ↑ Hier wird z.B. blog.antlers.html oder post.antlers.html eingefügt -->
</main>

7 Statamic — Das CMS

Statamic ist ein CMS (Content Management System), das auf Laravel aufbaut. Es speichert Inhalte als einfache Markdown/YAML-Dateien — keine Datenbank nötig. Änderungen am Inhalt sind direkt als Dateien im Projektordner sichtbar.

Die wichtigsten Konzepte

BegriffWas es istIm Projekt
Collection Eine Gruppe zusammengehöriger Inhalte Die Blog-Posts sind eine Collection
Entry Ein einzelner Inhalt in einer Collection Jeder Blog-Post ist ein Entry
Blueprint Definiert welche Felder ein Entry hat Titel, Datum, Bild, Beschreibung, Text…
Template Das Antlers-HTML, das den Entry darstellt blog.antlers.html, post.antlers.html
Control Panel Die grafische Admin-Oberfläche Unter /cp — zum Inhalte pflegen

Im Control Panel einloggen

URL:       http://random-thoughts.test/cp
Benutzer:  coding@basics.ch
Passwort:  codingbasics
🎛️

Im Control Panel kannst du Blog-Posts erstellen, bearbeiten und die Felder-Struktur (Blueprint) konfigurieren — alles ohne Code anfassen zu müssen.

8 Statisch → Dynamisch

Das ist der Kernunterschied zwischen den zwei Projekt-Phasen. Im static-Branch sind alle Inhalte fix im HTML. Im main-Branch kommen sie dynamisch aus dem CMS.

Statisch — Branch: static
<!-- Jeder Post hartkodiert -->
{{ partial:components/card
    image="/assets/octopus.jpg"
    title="There's something
      secret and dangerous..."
    date="15. April 2025"
    url="#"
    class="md:col-span-2"
}}

<!-- 5x manuell wiederholt -->
Dynamisch — Branch: main
<!-- Loop über alle CMS-Posts -->
{{ collection:posts
    sort="date:desc" }}

  {{ partial:components/card
      image="{ cover }"
      title="{ title }"
      date="{ date | format:d. F Y }"
      url="{ url }"
  }}

{{ /collection:posts }}

Im dynamischen Ansatz iteriert Antlers automatisch über alle Einträge in der Collection posts und rendert für jeden Eintrag eine Card mit den echten CMS-Daten. Kommt ein neuer Post im Control Panel hinzu, erscheint er sofort auf der Blog-Seite — ohne Code-Änderung.

✨

Das ist der eigentliche Mehrwert eines CMS: Redakteure können Inhalte pflegen, ohne den Code zu kennen. Der Entwickler baut die Struktur einmal — der Rest passiert im Control Panel.

9 Installation Schritt für Schritt

So richtest du das Projekt lokal ein:

  1. Repository in den Herd-Ordner klonen
    Entweder in Tower («Clone») oder im Terminal:
    cd ~/Herd
git clone [repo-url] random-thoughts
  2. Projekt in VS Code öffnen
    code ~/Herd/random-thoughts
  3. .env Datei anlegen und URL setzen
    cp .env.example .env
# In .env dann ändern:
APP_URL=http://random-thoughts.test
  4. PHP-Abhängigkeiten installieren (Statamic, Laravel, etc.)
    composer install
  5. App-Schlüssel generieren (Sicherheits-Token für die App)
    php artisan key:generate
  6. Frontend-Pakete installieren (Tailwind CSS, Vite, Alpine.js)
    npm install
  7. Frontend-Build starten (CSS und JS werden kompiliert)
    npm run dev
🔄

npm run dev muss im Hintergrund aktiv bleiben während du entwickelst. Es überwacht deine CSS/JS-Dateien und kompiliert Änderungen automatisch. Ohne diesen Prozess sehen neue Tailwind-Klassen keine Wirkung.
